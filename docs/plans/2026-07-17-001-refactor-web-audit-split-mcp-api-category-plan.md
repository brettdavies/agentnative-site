# refactor: split the web-audit MCP & API category into separate API and MCP categories

## Summary

The web-audit registry groups MCP checks and API checks under one `mcp-api` category ("MCP & API"). Split it into two
categories so the board and scorecard show them apart. Target category order: **C1 Discoverability, C2 Content for
agents, C3 Bot & crawl policy, C4 API, C5 MCP, C6 Agent discovery & auth**. This is a display re-grouping only:
per-check `tier` and `weight` are unchanged, so `universeMax`, the global/relative scores, and the board ordering do not
move. Retiering `openapi` (the one lever that would change scoring) is explicitly deferred to a follow-up.

---

## Problem Frame

`mcp-api` conflates two distinct agent surfaces. An MCP server and an OpenAPI/JSON-Schema API are audited by different
checks and remediated differently, but the scorecard rolls them into one card and the homepage shows one row, so a
reader cannot see "strong MCP, no API" (or the reverse) at a glance. Splitting the category makes each surface a
first-class, separately-scored-visible group without touching how points are earned.

**Scope decision (confirmed):** land the pure display split now with both new categories at their existing display tier
(MUST); reconsider `openapi`'s check tier in a separate PR so a scoring change is never entangled with the
re-categorization.

---

## Requirements

- **R1.** The registry exposes six categories in order: `discoverability`, `content-for-agents`, `bot-crawl-policy`,
  `api`, `mcp`, `agent-discovery-auth`. `mcp-api` no longer exists.
- **R2.** API checks (`openapi`, `json-schemas`, `api-catalog`) carry `category: api`. All MCP checks (`mcp-initialize`,
  `mcp-capabilities`, `mcp-tools-list`, `mcp-unknown-method`, `mcp-get-fast-fail`, `mcp-cors-preflight`,
  `mcp-cors-actual`, `well-known-mcp-card`, `mcp-usage-doc`, `webmcp`) carry `category: mcp`.
- **R3.** The scorecard renders six category cards, numbered C1-C6 in the order above, with display tiers: API = MUST,
  MCP = MUST, agent-discovery-auth = MAY (all others unchanged).
- **R4.** The homepage web-check pane shows six rows matching the six categories (C4 API, C5 MCP, C6 Agent discovery &
  auth).
- **R5.** Scoring is unchanged: for any given site, `universeMax`, the global score, and the relative score are
  identical before and after the split. Board ordering is unchanged.
- **R6.** The web-scorecard-schema doc reflects the six categories.

---

## Key Technical Decisions

- **KTD1 — Display split, not a scoring change.** `score.ts` computes `universeMax` and earned points per check
  (`universeMaxOf`, `scoreWebAudit`); `categoryRollups` only groups results by `category_order` for display. Reassigning
  a check's `category` moves which card it renders under but not its tier/weight, so scores are invariant. This is the
  whole reason the split is safe to ship without a rescore-visible score shift.
- **KTD2 — The scorecard C-index is derived, the homepage's is hardcoded.** `summary-render.ts` numbers cards by
  iterating `scorecard.categories` (built from `category_order`), so C1-C6 follows automatically once the registry
  changes. `src/build/06-homepage.mjs` `WEB_CHECKS` is a static array with literal `C1`-`C5` ids and must be edited by
  hand to add the sixth row and renumber. These are the only two renderer surfaces.
- **KTD3 — API display tier = MUST for now.** `openapi` is a `required` check today, so the API card inherits MUST. The
  question of softening API to SHOULD is a check-tier change (scoring-affecting) and is deferred (see Scope Boundaries).
- **KTD4 — Category display names.** `api` -> "API", `mcp` -> "MCP", `agent-discovery-auth` -> "Agent discovery & auth"
  (unchanged). Category ids are lowercase-kebab to match the existing convention.

## Implementation Units

### U1. Registry re-categorization

**Goal:** Replace the `mcp-api` category with `api` and `mcp` in the registry source, reorder, and reassign each check's
`category`.

**Requirements:** R1, R2.

**Dependencies:** none.

**Files:**

- `src/data/web-audit/registry.yaml` (modify: `category_order`, `categories` map, and the `category:` field on the
  thirteen affected checks)
- `dist/_internal/web-audit-registry.json` (regenerated by `bun run build`; not hand-edited)

**Approach:**

- `category_order`: `[discoverability, content-for-agents, bot-crawl-policy, api, mcp, agent-discovery-auth]` (api
  before mcp per C4/C5).
- `categories`: drop `mcp-api: MCP & API`; add `api: API` and `mcp: MCP`.
- Set `category: api` on `openapi`, `json-schemas`, `api-catalog`; `category: mcp` on the ten MCP checks listed in R2
  (including `webmcp`).
- Optionally reorder the check blocks and the section comment so the file reads API-then-MCP, matching the new order
  (cosmetic; the projection sorts by `category_order`, not file order).

**Test scenarios:**

- Registry-walk coverage: every check's `category` is a key in `categories`; `category_order` is exactly the set of
  `categories` keys in order; no check references `mcp-api`.
- Category-order drift: the projected `dist/_internal/web-audit-registry.json` `category_order` equals the six-id list
  in R1.
- Covers R1, R2.

**Verification:** `bun run build` regenerates the registry JSON with six categories; a grep for `mcp-api` across `src/`
and `dist/_internal/` returns nothing.

## Scope Boundaries

### In scope

- Registry category definitions, order, and per-check `category` reassignment.
- Scorecard renderer display tiers + the homepage web-check rows.
- Schema doc + drift/invariance tests.

### Deferred to follow-up work

- **Retier `openapi` (API MUST -> SHOULD).** Downgrading `openapi` from `required` to `recommended` would make the API
  category read SHOULD and would change `universeMax` and every seeded score, reflowing the board on the next rescore.
  Kept out of this PR so the re-categorization ships with zero scoring movement; open a separate `feat`/`refactor` PR to
  decide it on its own merits.

### Out of scope

- Any change to which checks exist, their weights, antecedents, or handlers.
- Board ranking, the friendly-name display, or the rescore Workflow (all shipped in PR #224).

---

## System-Wide Impact

- **Read surfaces:** `/web/<domain>` scorecard (HTML + `.md`) and the homepage web-check pane change grouping/labels.
  The `/web` board and its ordering are unaffected (board renders scores, not categories).
- **Stored data:** per-domain scorecards in R2 carry `results[].category`. Existing cached scorecards still hold
  `category: mcp-api`; they render under the fallback until re-audited. A rescore after merge rewrites them with `api`/
  `mcp`. No migration needed; the next weekly cron (or a deploy-hook rescore) reflows them. Note this in the PR so the
  transient mixed state is expected.
- **Schema version:** `WEB_SCHEMA_VERSION` (0.2) need not bump — the shape is unchanged, only category enum values. The
  spec `SPEC_VERSION` (0.5.0) is unrelated and unchanged, so R2 keys do not rotate.
