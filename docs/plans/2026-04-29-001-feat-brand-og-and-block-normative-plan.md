# feat: Brand-aligned OG image + block-level MUST/SHOULD/MAY treatment

## Overview

Two paired post-v0 brand polish items, intentionally bundled because they share the same skill chain and visual context.

**Item A — Block-level normative treatment.** Today, `MUST` / `SHOULD` / `MAY` keywords render as inline color-coded
spans inside flowing paragraphs (handled by `src/build/plugins/rfc-keywords.mjs`). Each principle page also has a
predictable shape: a `## Requirements` section containing `**MUST:**`, `**SHOULD:**`, and `**MAY:**` paragraphs, each
immediately followed by a bulleted list. The shape is consistent across all seven principle files. This plan promotes
that exact pattern — `**KEYWORD:**`-paragraph-immediately-followed-by-list — into a semantic block container with a
keyword badge, while leaving every other inline keyword usage untouched.

**Item B — OG image regeneration.** `public/og-image.png` was generated by `scripts/og/generate.py` against
`gemini-3-pro-image-preview` during M1 scaffold work (commit `743b8fc`, 2026-04-14) and is now stale relative to the
typography calibration that landed in todo `006`. More importantly, the current image is fully monochrome — it does not
carry the site's strongest visual signature, the `MUST` / `SHOULD` / `MAY` color trio. This plan replaces the LLM image
generator with a deterministic Playwright HTML→PNG renderer that uses the site's actual self-hosted fonts and the same
`docs/design/foundation.css` color tokens, then redesigns the card itself through a structured design-skill chain.

The two items ship as **two separate PRs into `dev`** per the user's `feedback_dev_branch_squash_flow.md` memory: PR 1 =
Item A (Unit 1), PR 2 = Item B (Units 2-5). Item A ships first because block-level rendering changes any future site
screenshot and the OG card is itself a screenshot of typeset HTML.

This plan is the deliverable for the originating session and is intended to be executable from a **cold session** with
no prior conversation context. Every decision has been resolved in writing below.

## Key Technical Decisions

### Item A — Block-level normative treatment

- **Decision: Promote `**KEYWORD:**`-paragraph-followed-by-list, leave everything else inline.** This is the rule
  authors can apply (R1). It maps exactly onto the shape every principle file already uses. Mid-paragraph keyword
  qualifiers stay inline — the existing `rfc-keywords.mjs` plugin keeps owning that.
- *Rationale:* Avoids origin doc Option 2's heuristic-surprise risk (a paragraph that *starts* with a keyword but is not
  a normative declaration stays inline). Avoids origin doc Option 3's plugin-dependency cost (no `remark-directive`).
  Achieves origin doc Option 1's clarity without per-instance author markup, because the markdown source already encodes
  the intent.
- **Decision: Implement as a new mdast-level remark plugin, not a rehype-level transform.**
- *Rationale:* Mdast nodes still distinguish `paragraph` and `list` cleanly. Rehype-level requires reasoning about `<p>`
  and `<ul>` as siblings under `<section>` after `remark-rehype` already separated them, which is more fragile.
- **Decision: Run the new plugin AFTER `rfc-keywords` in the unified pipeline.**
- *Rationale:* The block container wraps the inline-styled paragraph and list. By running second, the inner `<strong
  class="rfc-must">MUST</strong>` is already in place when the wrapper is added.
- **Decision: Render container as `<aside class="normative normative--must" aria-labelledby="…">` with a leading badge
  child.** Single semantic element, ARIA-labelled by the badge. Color-not-only differentiation (R8) provided by the
  badge text itself (`MUST` / `SHOULD` / `MAY`) plus a small geometric difference (badge uses tabular figures
- spec-style tracking — exact treatment locked in by `/typeset`).

- **Decision: Markdown source unchanged.** R4 resolved by construction. The `/p1.md` etc. endpoints continue to serve
  identical bytes; agents see `**MUST:**\n\n- …` exactly as before.

### Item B — OG image regeneration

- **Decision: Replace `gemini-3-pro-image-preview` with deterministic Playwright HTML→PNG.** The existing Python
  generator at `scripts/og/generate.py` is deleted in Unit 4.
- *Rationale:* (a) Eliminates glyph drift — the actual `Uncut Sans` and `Monaspace Xenon` woff2 files render the text
  via Chromium's font engine instead of being approximated by an LLM. (b) Reproducible: same inputs always produce
  identical output bytes, so a future renderer change shows up as an obvious binary diff. (c) Zero API cost, no
  `GEMINI_API_KEY` required. (d) Tracks future design changes automatically because it consumes
  `docs/design/foundation.css` for color tokens. (e) Playwright is already a dev dep; no new dependencies.
- **Decision: HTML/CSS for the card lives at `docs/design/og.html` + `docs/design/og.css`.** Reusable for further
  iteration without touching the generator script. The `og.html` references local font URLs and the same
  `foundation.css` shipping to production.
- **Decision: Concept exploration via `/design-shotgun` before any HTML is written.** Three concepts — A spec-fragment /
  B publication-TOC / C single-statement — were sketched in the originating session. The skill produces all three as
  renderable variants and a comparison artifact; the user picks one before Unit 3.
- *Rationale:* The originating session conversation explicitly raised brand fit as the open question. Skipping shotgun
  would commit to one direction on faith.
- **Decision: Finalization via `/impeccable` (not `compound-engineering:frontend-design`).** The repo already declares
  `.impeccable.md` as the project's design context file (the file frontmatter explicitly lists `/impeccable`, `/typeset`
  as consumers); `docs/DESIGN.md` references the impeccable skill's `<absolute_bans>` list as authoritative. Picking the
  matching skill family keeps the AI-slop bar consistent with what shipped on the site.
- **Decision: Typography polish via `/typeset`, visual QA via `/design-review`.** Required by the user's
  `feedback_route_visual_fixes_via_skills.md` memory and by R5 / R12.
- **Decision: Add `og:image:alt` meta tag to `src/build/shell.mjs`** — currently missing (R10). Single-source the alt
  text in `shell.mjs` so it applies to every page (matching the current single-OG model).
- **Decision: Generator runs as `bun run og` script, not on every build.** Output is committed binary at
  `public/og-image.png`; `assets.mjs` continues to copy it byte-for-byte to `dist/og-image.png`. Running on every build
  would create unnecessary binary diffs in PRs.
- *Rationale:* Matches the current discipline (DESIGN.md §4.13: "Generated out-of-band … not regenerated on every
  build"). Locks the asset's contents to commits where the generator output is actually exercised.
- **Decision: PNG optimization via Sharp's PNG palette quantization, not pngquant.** Avoids adding a binary toolchain
  dependency (the current Python script uses Pillow's quantize for the same reason).
- **Alternative if Sharp underperforms:** call out to `pngquant` if Homebrew-installed locally, fall back to Sharp. Keep
  the budget at <100 KB (current image is 67 KB; new image is acceptable up to ~150 KB per origin doc).

## High-Level Technical Design

> *This illustrates the intended approach and is directional guidance for review, not implementation specification.
> The implementing agent should treat it as context, not code to reproduce.*

### Item A — block transform shape (mdast-level)

```text
Input mdast (after rfc-keywords plugin already ran):
  paragraph
    strong[children=text("MUST:")]            ← already class=rfc-must from rfc-keywords plugin
  list (ordered=false)
    listItem … listItem …

Output mdast (new plugin transforms in place):
  containerDirective | aside-shaped node      ← hName=aside, hProperties={class=[normative, normative--must]}
    paragraph                                  ← preserved verbatim from input
      strong.rfc-must                          ← unchanged inline span
        text("MUST:")
    list                                       ← preserved verbatim from input
      listItem … listItem …

Detection rule (executed for every paragraph node):
  IF paragraph.children is exactly [strong{rfc-must|rfc-should|rfc-may}]
     OR [strong{rfc-must|rfc-should|rfc-may}, text(":")]      ← in case rfc-keywords annotated only the keyword
  AND the paragraph's right-sibling in its parent is a `list` node
  THEN wrap [paragraph, list] in a new aside-shaped node and replace both originals with it.
  ELSE leave both nodes untouched.
```

The detection is conservative: it requires the paragraph to contain *only* the colon-suffixed keyword (no other prose),
and requires the very next sibling to be a list. This matches every existing principle file's shape and nothing else.

### Item B — Playwright→PNG pipeline

```text
docs/design/og.html
  ├─ <link rel="stylesheet" href="../../docs/design/foundation.css">  ← color tokens
  ├─ <link rel="stylesheet" href="./og.css">                          ← layout + composition
  └─ @font-face → file://…/public/fonts/{uncut-sans,monaspace-xenon}-variable.woff2

scripts/og/generate.ts
  1. read VERSION literal from src/build/shell.mjs (regex match SITE_TAGLINE block)
  2. inject into og.html via document.querySelector('[data-version]').textContent
  3. launch Playwright Chromium, viewport: 1200x630, deviceScaleFactor: 2
  4. page.goto('file://…/docs/design/og.html'); wait for fonts via document.fonts.ready
  5. page.screenshot({ path: '/tmp/og-2x.png', clip: {0,0,1200,630} }) → 2400x1260 PNG
  6. Sharp: resize to 1200x630, palette-quantize, write public/og-image.png

package.json scripts:
  "og": "bun scripts/og/generate.ts"

CI:
  No CI step. Generator runs locally; PNG is committed.
```

**Approach:**

- Mdast plugin walks `paragraph` nodes. For each paragraph that contains exactly one child of type `strong` whose
  data.hProperties.className includes one of `rfc-must` / `rfc-should` / `rfc-may` AND ends with `:` (either as the
  strong's text or as a trailing `text` sibling), AND whose parent's next sibling is a `list` node — replace the
  [paragraph, list] sibling pair with a new node that has `data.hName = 'aside'` and `data.hProperties.className =
  ['normative',`normative--{must|should|may}`]`. Children are `[paragraph, list]` preserved verbatim.
- Run the visit pass single-direction (top-down with index tracking, replace via parent.children.splice) — same pattern
  as the existing `rfc-keywords.mjs` lines 100-104.
- CSS: match existing site.css inline-keyword conventions. Container is full-width within the prose measure
  (`max-inline-size: var(--measure)`). Padding/border styles route through `/typeset`. Anti-pattern reminder:
  `border-left > 1px` is the impeccable skill's #1 banned pattern (cited in `docs/DESIGN.md`); do NOT use a side stripe.

**Execution note:** Test-first — write the new test cases in `tests/build.test.ts` *before* the plugin implementation.
Each test should assert the rendered HTML structure for a representative input. The unit's correctness is entirely about
HTML shape, which is naturally test-led.

**Technical design:** See "High-Level Technical Design" §A above. Detection rule is conservative by intent —
mid-paragraph or non-list-followed normative keywords stay inline.

**Patterns to follow:**

- `src/build/plugins/rfc-keywords.mjs` — visitor pattern, parent index tracking, replacement application back-to-front.
- `tests/build.test.ts` lines 61-130 — `rfc-keywords plugin` describe block style.
- `docs/design/foundation.css` lines 210-212 — token usage convention (consume `--must` / `--should` / `--may`
  variables; never hardcode hex).

**Test scenarios:**

- *Happy path — MUST:* Markdown `\n**MUST:**\n\n- item one\n- item two\n` produces an `<aside class="normative
  normative--must">` containing both the paragraph (with the existing `<strong class="rfc-must">` inside) and the
  `<ul>`. Single aside in the output.
- *Happy path — SHOULD and MAY:* Same as MUST with `should` and `may` class variants.
- *Edge case — non-block-shaped MUST stays inline:* Markdown `Tools MUST run non-interactively when stdin is not a TTY.`
  produces output that does NOT contain `<aside`, but DOES contain `<strong class="rfc-must">MUST</strong>` exactly
  once. Mid-paragraph keywords are unchanged from current behavior.
- *Edge case — `**MUST:**` paragraph NOT followed by a list stays inline:* Markdown `**MUST:**\n\nNo list here, just
  prose.` produces no `<aside` and the paragraph still carries its inline `<strong class="rfc-must">MUST:` treatment.
- *Edge case — MUST NOT inside a block:* Markdown `**MUST NOT:**\n\n- item` (hypothetical; not currently in the corpus)
  produces an aside with `normative--must` (the negation does not change the class — it carries the same visual weight
  as MUST, matching how `rfc-keywords.mjs` already maps `MUST NOT → rfc-must`).
- *Edge case — keyword is NOT colon-suffixed:* Markdown `**MUST**\n\n- item` (bold MUST without trailing colon) does NOT
  promote — the rule requires `:`. This guards against accidentally lifting a one-word emphasis.
- *Integration — every principle file:* For each of `content/principles/p[1-7]-*.md`, `renderMarkdown(file)` produces
  2-3 normative asides matching the `^\*\*(MUST|SHOULD|MAY):\*\*` regex matches we counted in the originating session
  (P1: 3 asides, P2: 3, P3: 3, P4: 2, P5: 2, P6: 3, P7: 3 — total 19).
- *Integration — preserves inline plugin behavior inside the block:* Asserting the inner `<p>` of the aside contains
  `<strong class="rfc-must">MUST:</strong>` exactly once (no nested `<strong><strong>` regression of the prior bug fixed
  in `rfc-keyword-remark-plugin-nested-strong-2026-04-14.md`).
- *Integration — markdown bytes unchanged:* For each principle file, `read(p1.md)` returns the same source after this
  change as before (verified by hashing the file before and after — sanity check that we didn't accidentally rewrite
  source).
- *Accessibility — color-not-only differentiation (R8):* The aside's badge contains the literal text `MUST` / `SHOULD` /
  `MAY`, so a stylesheet-disabled render (or a print stylesheet, or a screen reader) still conveys the keyword identity
  textually.

## System-Wide Impact

- **Interaction graph:** Item A modifies the unified markdown→HTML pipeline (`src/build/render.mjs`). The new plugin
  runs once per principle page render. No new runtime dependencies beyond what `unist-util-visit-parents` (already a
  transitive) provides. Item B introduces a new offline asset-build step, isolated from the production build pipeline.
- **Error propagation:** New plugin must not throw on unexpected mdast shapes — log and pass through the original
  paragraph/list. Generator script exits non-zero on any failure (font load, render, write); operator runs locally and
  notices.
- **State lifecycle risks:** Item A is pure transformation (no persisted state). Item B writes a single binary file to
  disk; idempotent. No partial-write risk because Sharp writes via tmp + atomic rename by default.
- **API surface parity:** The markdown endpoint (`Accept: text/markdown` content negotiation, `.md` URL suffix) must
  continue to serve identical bytes. R4 verification covers this.
- **Integration coverage:** All seven principle pages must render the expected number of normative asides (see Unit 1
  test scenarios). The aggregate count (19) is the integration-coverage signal.
- **Unchanged invariants:**
- The existing `rfc-keywords` mdast plugin keeps owning inline keyword annotation. Mid-paragraph keywords behave exactly
  as today.
- The markdown source files in `content/principles/` are unmodified.
- The OG meta tag set keeps its current shape; only `og:image:alt` and `twitter:image:alt` are added.
- The asset's filename and path (`public/og-image.png` → served as `/og-image.png`) is unchanged. Cache headers
  (immutable per `src/worker/headers.ts:54`) need no update — replacing the file's contents while keeping the URL means
  CDN purge happens automatically on deploy.

  lands. The OG generator reads from this same literal so OG and footer never drift.
